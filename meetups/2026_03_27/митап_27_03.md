## 1. Задачи недели
Главной и основной задачей прошедшей недели стало прототипирование.
Работа была продолжена сразу в нескольких направлениях, связанных с разработкой архитектурных механизмов взаимодействия приложения с удалённым репозиторием, файловой системой и пользовательским интерфейсом.
Для удобства проектирования и реализации все задачи были условно разделены на две основные группы. Первая группа связана с пользовательским интерфейсом: его структурой, логикой работы и отображением данных. Вторая группа относится к backend-части приложения и включает вопросы взаимодействия с удалённым репозиторием, а также организацию внутренней логики работы с git-функционалом.

Сделано:
- продуман конкретный прототип GUI: проанализированы основные пользовательские сценарии, выделен минимально необходимый на текущем этапе функционал, а также определена логика взаимодействия интерфейса с файловой системой, включая её организацию и соотношение со структурой интерфейса;

- проведён анализ библиотек для интеграции git-функционала - выбрана libgit2;

- начата практическая реализация прототипов GUI и backend-компонентов. Со стороны пользовательского интерфейса начата разработка основного окна программы, предназначенного для выбора и отображения скриптов. Со стороны backend-части начата разработка ядра для работы с git, которое в дальнейшем будет обеспечивать API для изолированной и структурированной работы с удалённым репозиторием.

## 2. Архитектурные пояснения

Архитектура программы строится по принципу разделения ответственности. Графический интерфейс не взаимодействует напрямую с локальной файловой системой, git-функционалом и удалённым репозиторием. Вместо этого каждая подсистема изолируется в отдельный компонент, имеющий собственный интерфейс взаимодействия. GUI передаёт запросы в прикладную часть программы, прикладная часть обращается к обёрткам работы с локальной файловой системой и удалённым репозиторием, а результаты их работы возвращаются обратно в GUI в уже подготовленном для отображения виде.
Плюсы такого подхода:
- компоненты не перегружаются;
- компоненты независимы друг от друга;
- управляемая интеграция.

 |-- компонент извлечения метаданных из скриптов
 |-- компонент работы с git/удалённым репозиторием


## 3. Прототип GUI
Начнем со структуры и логики GUI:
Было принято решение добавить в программу два режима: базовый и кастомный. Базовый режим ориентирован "рабочие" скрипты - такие скрипты наиболее вострбованы на производстве - поступающими из удалённого репозитория. Пользователь выбирает свою роль, затем прибор, после чего интерфейс отображает только доступные скрипты. При этом список формируется не вручную, а на основе метаданных, встроенных в сами скрипты. Дополнительно рядом с названием скриптов могут отображаться их статусы, например новый, изменённый или конфликтующий - это статусы поступающие с backend-части. В этом же режиме предусмотрен минимальный набор базовых действия: выбор скрипта, запуск, открытие в редакторе.
Идея кастомного же режима заключается в том, чтобы предоставить пользователю личное рабочее пространство. В этом режиме работа идёт уже с локальными файлами, которые не попадают в удалённый репозиторий. Кроме того, здесь вводится механизм пресетов, то есть пользовательских наборов скриптов, которые можно формировать под конкретные задачи. Система становится более гибкой, поскольку позволяет объединять в одну группу разные скрипты независимо от их стандартной классификации. Сейчас реализован прототип внешнего интерфейса начального окна обоих режимов, в дальнейшем оно будет дорабатываться, расширяя функционал.

## 4. Библиотека libgit2
В качестве библиотеки для работы с git-функционалом была выбрана libgit2 - это библиотека на языке C, которая реализует основную функциональность и позволяет встроить работу с Git прямо в приложение, без вызова внешней консольной утилиты. Она даёт программный доступ к таким операциям, как открытие и создание репозитория, клонирование, работа с коммитами, анализу состояния репозитория и др.
Также существуют другие библиотеки и обёртки, например JGit для Java, LibGit2Sharp для .NET, pygit2 для Python и NodeGit для Node.js. Однако многие из них либо привязаны к конкретному языку, либо сами строятся поверх libgit2.
Именно libgit2 в данном случае удобно использовать потому, что она хорошо подходит для встраивания git-механизмов в собственное приложение, работает как отдельная библиотека, поддерживает кроссплатформенный сценарий и позволяет построить поверх неё собственное ядро с понятным API. 

## 5. Ядро для git
Его основная задача состоит в том, чтобы вынести всю git-логику в самостоятельный низкоуровневый компонент и не допустить, чтобы интерфейс или другие части программы напрямую работали с библиотекой libgit2. Иными словами, ядро должно предоставить приложению собственный внутренний API для работы с репозиторием, а все детали конкретной реализации скрыть внутри себя. Ядро было решено реализовывать на языке C, поскольку используемая библиотека libgit2 предоставляет именно C-интерфейс. За счёт этого ядро может напрямую работать с её функционалом без лишних промежуточных слоев.
По файлам структура следующая. core.h — это общий заголовочный файл, через который ядро подключается к остальной системе. git_repo.h содержит основные функции для работы с репозиторием. git_base.h включает базовые низкоуровневые функции, которые уже напрямую опираются на libgit2. git_err.h отвечает за централизованную обработку ошибок, чтобы не дублировать эту логику в разных частях кода.
Если говорить по функциям, то в ядре предполагаются функции для инициализации работы с Git, открытия репозитория, получения информации о файлах, определения их состояния, обработки ошибок и другой базовый функционал как:pull, push, fetch, commit и тд. Пока реализовано: инициализация локального репозитория, подключение к существующему и клонирование удаленного репозитория.
